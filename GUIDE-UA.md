# SDD-плагін для команди Integrator — гайд

Командний плагін для Claude Code, який веде фічу за методологією **Spec-Driven Development**:
від ідеї → специфікації → дизайну → задач → коду → рев'ю → PR. Кожен етап читає артефакт
попереднього і **жорстко відмовляється працювати**, якщо його немає, — тож пропустити крок не вийде.

Плагін відкалібрований під Prime.Integrator: PHP 8.5 / Laravel 13, патерн
Client→Authenticator→Mediator, правила комітів `[TASK-JIRA]`, документація в репо (`docs/`).

---

## Встановлення

Репо приватне (тестове, на `SashaCN`) — потрібен git/SSH-доступ до нього. У Claude Code:

```
/plugin marketplace add SashaCN/claude.integrator_sdd
/plugin install sdd@integrator-sdd
```

Перевірити, що став: у новій сесії введи `/` — мають з'явитися команди `survey`, `specify`, `design` тощо.
Команди можна звати як `/survey`, так і повним `/sdd:survey` — це те саме.

**Оновлення:** коли вийде нова версія — `/plugin marketplace update integrator-sdd`.

## Як це працює — основний конвеєр

Робота йде по гейтованому «хребту». Типовий шлях фічі:

| # | Команда | Що робить | Артефакт на виході |
|---|---------|-----------|--------------------|
| 0 | `/survey` | Сканує репо → карта архітектури (або закладає фундамент у порожньому проєкті) | `docs/architecture-map.md` |
| 1 | `/specify {slug}` | Socratic-інтерв'ю → специфікація фічі (цілі, історії, критерії приймання, NFR) | `docs/features/{slug}/spec.md` |
| 2 | `/clarify {slug}` | Полювання на неоднозначності в спеці — закриває все, де два інженери зробили б по-різному | оновлений `spec.md` |
| 3 | `/design {slug}` | Архітектурний документ (Arc42 + C4), ADR на рішення з великим «радіусом» | `docs/features/{slug}/sad.md` (+ `docs/adr/`) |
| 4 | `/sequences {slug}` | Діаграми послідовності критичних потоків (happy + error) | секція §6 у `sad.md` |
| 5 | `/data-model {slug}` | Модель даних + **готові** міграції up/down | `data-model.md` + `migrations/` |
| 6 | `/api {slug}` | Контракт API (OpenAPI 3.1) — виводиться з моделі та діаграм | `contracts/openapi.yaml` |
| 7 | `/tasks {slug}` | Розбиття на атомарні задачі (≤1 день) з графом залежностей | `tasks.json` + `tasks/` |
| 8 | `/plan-tests {slug}` | Тест-план: кожен критерій приймання → щонайменше один тест | `test-plan.md` |
| 9 | `/implement {slug}` | TDD-реалізація задач (тест → код → рефактор → гейт), один коміт на фічу | код + тести |
| 10 | `/review {slug}` | Незалежне код-рев'ю в чистому контексті проти спеки | звіт рев'ю |
| 11 | `/ship {slug}` | Перевіряє, що фіча реально працює, пише changelog, готує PR | PR |

> `{slug}` — коротка назва фічі (напр. `bulk-usage-sync`). Не всі кроки обов'язкові для дрібних
> фіч — розмір регулюється (див. `/classify-size`).

Над окремими фічами живе **роадмеп портфеля**: `/roadmap` (Now / Next / Later). `specify` піднімає
фічу в Now, `ship` переносить у Shipped — роадмеп лишається синхронним сам собою.

## Допоміжні команди (поза хребтом)

| Команда | Призначення |
|---------|-------------|
| `/classify-size {slug}` | XS/S/M/L/XL — скільки артефактів реально потрібно фічі |

| `/decide-adr` | Записати архітектурне рішення постфактум (MADR ADR) |
| `/document {slug}` | Синхронізувати durable-доку (`docs/`) з реальністю після фічі |

## Що варто знати

- **Гейтинг.** Кожен крок вимагає артефакт попереднього. Немає `spec.md` → `design` відмовиться. Це навмисно.
- **Handoff.** Наприкінці кожного етапу плагін друкує блок: що змінилось, які файли глянути, далі — `/clear` і наступна команда. **Роби `/clear` між етапами** — це тримає контекст чистим.
- **Глибина інтерв'ю.** Q&A-скіли (`specify`, `clarify`, `design`) налаштовуються: `easy` (вирішує за тебе) → `hard` (допитує з трейд-офами + запускає аналітиків-субагентів: дослідник ринку, стратег, devil's advocate).
- **Документація.** Durable-доки (`docs/architecture/`, `docs/ecosystem/`, `docs/standards/`, `docs/adr/`) комітяться; процесний слід (`docs/features/`, `docs/architecture-map.md`) — gitignored і одноразовий.
- **Survey = кеш, не істина.** `architecture-map.md` — згенерований індекс; джерело правди — `AGENTS.md` і `docs/`. При конфлікті виграють curated-доки, мапа перегенеровується.

## Типовий старт на новому проєкті

```
/survey                      # один раз — карта репо
/specify my-feature          # описати фічу
/clarify my-feature          # прибрати неоднозначності
/design my-feature           # архітектура
… далі по таблиці до /ship
```

Між кроками — `/clear`, потім наступна команда з handoff-блоку.
